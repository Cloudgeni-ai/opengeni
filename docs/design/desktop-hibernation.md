# Desktop Hibernation Across SaaS Sandbox Providers — Definitive Briefing (June 2026)

The goal: a cloud **desktop** sandbox that **hibernates** — on idle scale-to-zero, then resume with the *internal process/RAM state continuous* (X/Xvfb/x11vnc/noVNC + WM still running, browser process tree + live tabs + JS heap, terminals, dev servers still listening, in-memory app/DB state). External clients (noVNC socket, external DB/WS) reconnecting is acceptable. This is **memory/process preservation, not filesystem-only**. CRIU is the generic mechanism; the question is whether providers' *native* pause/resume primitives deliver CRIU-class continuity for a desktop.

## 2. E2B deep recommendation

**What pause/resume gives a desktop sandbox.** E2B's pause is a **CRIU-class full memory snapshot of the microVM**, not Chrome session-restore. Per the current first-party doc (e2b.dev/docs/sandbox/persistence): on pause "both the sandbox's filesystem and memory state will be saved … all the files … and all the running processes, loaded variables, data, etc."; on resume "all the running processes, loaded variables, and data are restored." Mapped to the goal's desktop items:
- **X/Xvfb + XFCE + x11vnc + noVNC/websockify** — ordinary Linux processes inside the microVM → captured in the RAM snapshot → come back running (the `nc -z` port readiness probe is the right signal; see §3).
- **Browser process tree + open tabs + JS heap** — **strong inferred case for true in-memory survival**: renderer PIDs and JS-only heap are in the RAM snapshot, so the *same* renderer process with the *same* heap should return. This is **NOT documented as a worked example** — high-confidence-from-mechanism, untested-by-docs (the headline thing to validate in §4).
- **Terminals/shell, in-memory app/DB, loaded variables** — explicitly covered.
- **Dev servers + listening ports** — survive; **confirmed empirically** by issue #1031 (a Next.js dev server stays alive and keeps holding port 3000 across auto-pause/resume).
- **Filesystem** — preserved.

**API (current, June 2026).** `Sandbox.create(...)` now carries the lifecycle config that used to be `beta_create`. Auto-hibernate-on-idle: create with `lifecycle.onTimeout` = pause (object form, `keepMemory: true` for a full memory snapshot vs filesystem-only) plus an idle `timeout`. Manual pause: `sandbox.pause()` (now with an optional `keepMemory`/`keep_memory` flag, added 2026-06-25 in `e2b@2.31.0` / py `2.30.0`). **Resume = `Sandbox.connect(sandboxId)`** — there is no `resume()`; connecting to a paused box auto-resumes it with full context. Teardown that deletes the snapshot: `sandbox.kill()`. Desktop stream: `desktop.stream.start()` / `getUrl()` (noVNC on 6080). The `beta_`-prefix concern in the source findings is now **largely resolved** — the create-time lifecycle is on stable `create()`; only "persistence" as a product carries the public-beta label.

**Cost / retention / latency.**
- **Retention: indefinite** — current first-party doc: "Paused sandboxes are kept indefinitely — there is no automatic deletion or time-to-live limit"; held until `kill()`. (The 30d/14d figures in older/secondary sources are **stale or about other platforms**.)
- **Cost:** while paused you stop paying RUNNING compute and pay only **storage** for the FS+memory snapshot (free allotment 10 GiB Hobby / 20 GiB Pro). The pricing page does not itemize a distinct paused-memory-snapshot line — confirm contractually for a multi-GiB desktop snapshot held long-term.
- **Latency:** pause ≈ **4 s / GiB** (a 4 GiB desktop ≈ 16 s); **resume ≈ 1 s**.

**What truly survives vs only-appears-to.** *Truly survives (memory snapshot):* X stack, renderer PIDs + JS-only heap, dev-server PID + listening port (confirmed #1031), terminal processes, in-memory state. *Only reconnects (by design, acceptable):* the external noVNC WebSocket and any external DB/WS — "all the clients will be disconnected … you need to connect clients again." *Known E2B-specific edges:* after `auto_pause` resume the dev server is alive and the **port is still held but the process is invisible to `commands.list()`** (#1031, open) — so **OpenGeni must not re-enumerate the X/browser/dev procs after resume; key readiness off ports, not the process list**. Also: a closed-but-recurring **multi-cycle filesystem bug** (#884) and a `connect()`-hardcodes-`autoPause:false` bug (#875) — **pin an SDK version and re-verify**.

## 3. OpenGeni integration path

**Yes — add an E2B desktop-hibernation backend, but as a *second lifecycle mode for the existing `e2b` backend*, not an 11th enum member.** OpenGeni already models suspend/resume in the contracts and already has resume-by-id — the gap is that **today's drain/re-warm cycle is FS-persistence + cold-recreate (`/workspace` tarball → fresh box), which destroys exactly the process/RAM state hibernation must keep.** The change is a descriptor-gated swap of the reaper's terminal action from *terminate* to *suspend*, plus treating *paused* as *resumable* not *gone*.

**Capability-descriptor fields** (`packages/contracts/src/index.ts`):
- Add a new `snapshot.kind: "native-suspend"` (or a `lifetime.suspendMode: "memory" | "filesystem"` discriminator) — the load-bearing new field: it tells the reaper "**pause, don't terminate**" and the resume path "**connect() restores processes; do NOT cold-recreate or re-run the X stack.**" (`index.ts:76-79`)
- `e2b`: keep `supportsSuspendResume:true`, keep `resumeIsLockFree:false` (E2B `connect()` needs the lease CAS to fence double-resume — no `fromId`-style lock-free reattach). (`index.ts:190-196`)
- **Flip `e2b.requiresSnapshotRollover` only if** you rely on a TTL — but **per current docs E2B retention is indefinite**, so unlike the source analysis's recommendation, **the 30-day-rollover requirement does NOT apply to E2B**; leave `requiresSnapshotRollover:false` unless you choose to roll over for hygiene. (This corrects the integration analysis, which proposed flipping it to `true` on the basis of a 30-day TTL that isn't real for E2B.)
- `hasIdleKiller:true` + set the hibernation profile's idle action to **pause** (config `e2bTimeoutAction:["pause","kill"]`, `packages/config/src/index.ts:369`) — now expressible as `lifecycle.onTimeout` with `keepMemory:true` on the current SDK.
- **Correct `blaxel.supportsSuspendResume → true`** (stale today; `index.ts:215-221`) — independent of the E2B work.

**Provider-seam work** (`packages/runtime/src/sandbox/providers/types.ts`, `providers/e2b.ts`): the seam is correct; the work is in the client verbs.
- `create` — unchanged; boot the `@e2b/desktop` image, pre-declare desktop+ttyd ports (`exposedPorts:[6080,7681]`, merged at `index.ts:264-289`; E2B is `preview-url` exposure).
- **Add a `suspend(state)` verb** = E2B `pause()`, distinct from `delete(state)` = `kill()`. **This is the one structural addition** — today the reaper only knows `delete`.
- `resume` → E2B `connect()` (already wired at `index.ts:730`'s `client.resume(state)` branch) — for hibernation this is the whole game.
- `serializeSessionState`/`deserializeSessionState` (`index.ts:316-319`, `:484-509`) carry the **paused-sandbox id**, NOT a `/workspace` archive. The `hasResumableInstance` gate (`index.ts:720-727`) already fires the warm-reattach branch on `providerState.sandboxId`. **Do NOT populate `sessionState.workspaceArchive`** (`index.ts:393`) — the memory snapshot *is* the persistence; a parallel tarball would trigger cold-recreate.
- `persistWorkspace` — **no-op/absent** for hibernation (the pause is the snapshot); the reaper's no-archive path (`sandbox-lease.ts:571-583`) then proceeds.

**noVNC-reconnect concerns (≈80% already there).** The reconnect plumbing exists and matches the "external clients reconnect — fine" model:
- Tunnel re-resolution is per-epoch: spawner records `data_plane_url` (`apps/worker/src/sandbox-resume.ts:419-441`); the slow path re-records a fresh `dataPlaneUrl` on epoch advance (`apps/api/src/sandbox/viewer.ts:433-458`); `resolveExposedPort(6080)` (`stream-port.ts:42-44`, called `:154`) re-resolves E2B's preview host post-resume.
- Viewer reconnect is event-driven: `mintDesktopStream` emits a **`stream.url.rotated` Channel-A event** on epoch advance (`viewer.ts:345-354`); the `StreamTokenPayload.leaseEpoch` fence (`contracts:681-695`) kills stale tokens.
- **The one design flip:** for hibernation, `ensureDisplayStack` must be a **verify-only no-op** — the X stack and browser tabs *survive the pause*, so re-launching it would kill the very tabs you hibernated. The fast `nc -z` port pre-check (`display-stack.ts:128-136`) already short-circuits when 6080/5900 are up, which is *also* the correct readiness probe given E2B's post-resume `commands.list` blindness (#1031). Make this an **assertion**, not an incidental behavior (`sandbox-resume.ts:359`).

**Idle-killer-vs-reaper (the real work).** The reaper is hardcoded to **terminate**; hibernation needs:
1. **`paused ≠ gone` discriminator** in `isProviderSandboxNotFoundError` (`index.ts:561-605`): the `goneMarkers` list includes `"no longer running"` — a *paused* box reporting "not running" would be **falsely cold-restored**, spawning a rival and discarding the hibernated state. Make `connect()` transparently un-pause (E2B `connect()` does resume a paused box), and add a `paused` arm if any provider surfaces "paused" as an error.
2. **Descriptor-driven `suspend()` vs `delete()`** in `terminateProviderBox` (`apps/worker/src/activities/sandbox-lease.ts:431-612`, esp. the `:588` delete): when `snapshot.kind==="native-suspend"`, call `suspend()`; skip `persistWorkspace`.
3. **New end-state `draining→suspended`** (or treat cold-with-resumable-id as suspended): the lease keeps `resumeState.providerState.sandboxId` pointing at a resumable paused box, so the next acquire hits the warm-reattach branch, not `coldRestore` (`index.ts:675-708`). The `hasResumableInstance` gate is already the right discriminator — the only fixes are the reaper not deleting and the cold-restore path not dropping the id.
4. **OpenGeni-driven pause is the recommended shape** (reaper calls `suspend()` at refcount 0, reusing the full drain-grace + epoch-fence + CAS), rather than provider-driven auto-pause, which would let the lease think the box is `warm` while E2B has paused it (state divergence + warm-second over-billing).

## 4. Concrete validation test plan

The experiment distinguishes **true memory hibernation** from **filesystem-restore / session-restore**. Run on **E2B first** (highest expected pass), then **Blaxel**, then **Daytona**.

**Setup (one desktop sandbox):**
1. Boot the provider's desktop image (E2B `@e2b/desktop`; Blaxel `cua-xfce`/`xfce-vnc`; Daytona `computer_use.start()`). Start the noVNC stream and connect a viewer.
2. **JS-memory-only counter tab:** in Chrome, open a page with `window._c = 0; setInterval(()=>window._c++, 1000)` — a value that exists **only in the renderer heap**, never persisted. Record `window._c` and the renderer PID (`pgrep -f "type=renderer"`).
3. **Logged-in tab:** authenticate to a site so a session cookie + in-memory auth state exist.
4. **Vite/Next dev server:** `npm run dev` on port 5173/3000; note its PID (`pgrep -f vite`) and confirm `curl localhost:5173` serves.
5. **Long-running terminal cmd:** `for i in $(seq 1 100000); do echo $i; sleep 1; done > /tmp/tick.log &` (or a `seq | while read` counter); note the PID and the current value.
6. Record X/x11vnc PIDs (`pgrep Xvfb`, `pgrep x11vnc`).

**Action:** Pause (E2B `pause()` / Blaxel drop-connection-15s-standby / Daytona non-force `stop()`/`pause()`), wait > the idle/snapshot window, then **resume** (E2B `connect()` / Blaxel reconnect / Daytona `start()`).

**Assertions — PASS only if ALL hold (true memory hibernation):**
| Check | PASS (memory hibernation) | FAIL (FS/session restore) |
|---|---|---|
| **Browser renderer PID** | Same PID as before pause | New PID (process re-spawned) |
| **JS-only counter** | `window._c` continues from its pre-pause value (and keeps ticking, having advanced ~by the paused wall-clock if timers resume, or frozen-then-continuing) | Resets to 0 / `undefined` (heap was lost) |
| **Tabs** | Live, never showed a crash/restore bar | Chrome "restore pages?" / reload spinner = session-restore, **FAIL** |
| **Dev-server PID** | Same PID, still alive | New PID or absent |
| **Dev-server port** | `curl localhost:5173` serves immediately, **no restart** | Connection refused until manually restarted |
| **Terminal cmd** | Same PID, counter advanced past its pre-pause value without restart | Process gone / log stops |
| **X/x11vnc PIDs** | Same PIDs; noVNC re-handshakes to the **existing** framebuffer (same window layout) | Fresh desktop / blank session |
| **Logged-in tab** | Still authenticated with no re-login | Logged out / re-auth |

**The single decisive discriminator:** the **JS-only counter + same renderer PID**. A filesystem/session-restore can bring back cookies, files, and a *relaunched* browser (FAIL on PID and on `window._c`); only a true RAM snapshot returns the **same renderer process holding the same heap** (PASS). The dev-server-port-without-restart and same-PID terminal are corroborating checks.

**Provider-specific watch-items during the run:**
- **E2B:** after resume, `commands.list()` may not show the dev-server/X procs (#1031) even though they're alive — **assert via ports/PIDs from a shell, not the SDK process list**. Run the multi-cycle FS check (#884): pause/resume **twice** and re-verify file mods persist. Pin the SDK version.
- **Blaxel:** test BOTH an image-baked/supervisord proc AND an **exec-launched** proc (the documented gap) — confirm both survive standby. Verify the noVNC preview URL is stable across standby, and that a viewer-less desktop doesn't standby mid-render (`keepAlive`/ping).
- **Daytona:** use **non-force** `stop()`/`pause()` (force clears RAM). This is the make-or-break test — docs don't promise desktop-daemon PID restoration and the VNC page says "start again before reconnecting." If renderer/X PIDs change → Daytona is **FS-restore for desktop**, not memory hibernation.

## 5. What could NOT be confirmed from docs — must be settled by the test

1. **The headline E2B desktop scenario is undocumented end-to-end.** That a paused *desktop* brings back the live x11vnc + Chrome with the **same renderer PID and same in-memory-only JS heap** (not Chrome session-restore) is **inferred from the memory-snapshot mechanism + corroborated only indirectly** (dev-server-survives, #1031). No E2B doc or blog demonstrates pause/resume of a live GUI desktop with open browser tabs. **This is the #1 thing the §4 test must prove.**
2. **Daytona Linux memory-on-pause for the desktop** — docs conflict ("stop clears memory" vs "stop behaves as pause") and never promise the **desktop daemon PIDs** survive; the VNC page implies a re-start. Whether pause→start restores the same X/browser tree is **unconfirmed**.
3. **Blaxel exec-launched-process restore** — docs guarantee "running processes" restore but never address procs started via the runtime exec API vs image-baked supervisord ones (Modal explicitly drops exec procs; Blaxel might not). Unconfirmed.
4. **Browser tab / JS-heap continuity** is claimed-by-inference for all three "process+RAM" providers (E2B, Blaxel, Daytona) — no provider has an explicit "browser tabs preserved" doc statement. Must be empirically verified per provider.
5. **Paused-state billing for the memory portion** of the snapshot is **not itemized** on any provider's pricing page (E2B/Blaxel/Daytona). Held-long-term multi-GiB memory-snapshot cost needs a contractual confirm.
6. **E2B retention "indefinite"** is what the current first-party doc says, but secondary sources disagree (stale 30d/14d figures) — confirm contractually for a multi-GiB snapshot held indefinitely. (Note: this is the **opposite** of what the OpenGeni integration analysis assumed; do not encode a 30-day E2B rollover requirement on the basis of the third-party blog.)
7. **GPU-backed desktops** — none of the providers document snapshotting a GPU desktop (E2B desktop is CPU/Xvfb; Modal explicitly bans GPU on snapshots). Out of scope unless GPU-accelerated Chrome is required.

# Desktop Hibernation in OpenGeni: Provider-Native Pause/Resume Integration Analysis

## Part 0 — Provider semantics, verified against current (June 2026) docs

The prior ranking is **mostly correct but stale in three specific places**. Corrections are flagged inline.

### E2B — confirmed best fit (prior analysis correct)
- Pause preserves **both filesystem and memory** — "all the running processes, loaded variables, data, etc." are restored on resume. ([e2b.dev/docs/sandbox/persistence](https://e2b.dev/docs/sandbox/persistence))
- Pause cost ~4s/GiB RAM; **resume ~1s**. SDK: `sandbox.pause()` / `sandbox.connect()` (resume) / `sandbox.kill()`.
- **Clients disconnect on pause and must reconnect on resume** — "all the clients will be disconnected… you need to connect clients again." This is exactly the noVNC-socket-reconnect model the goal accepts.
- **Retention correction:** docs are internally inconsistent — the persistence page says paused sandboxes are "kept indefinitely," but the current limits/pricing say **paused snapshots are deleted after 30 days** ([beam.cloud E2B pricing 2026](https://www.beam.cloud/blog/e2b-pricing-explained)). Treat it as a **30-day TTL on the paused snapshot**, not infinite. This matters for `requiresSnapshotRollover` (see Part 2).
- First-party **`@e2b/desktop`**: noVNC + websockify, `sandbox.stream.start()` + `sandbox.stream.getUrl()`. ([github.com/e2b-dev/desktop](https://github.com/e2b-dev/desktop))
- **Real caveat for desktop hibernation:** with `auto_paused=True`, after resume a dev server's port (e.g. 3000) **is still listening but the process is not enumerable via `commands.list`** ([E2B issue #1031](https://github.com/e2b-dev/E2B/issues/1031), [#884](https://github.com/e2b-dev/E2B/issues/884)). The X server / browser tree survive as *processes*, but OpenGeni must not rely on re-enumerating them after resume.

### Modal — confirmed weak for desktop hibernation (prior analysis correct)
[modal.com/docs/guide/sandbox-snapshots](https://modal.com/docs/guide/sandbox-snapshots), [memory-snapshots](https://modal.com/docs/guide/memory-snapshots):
- Memory snapshot = "exact clone… all running processes will still be running, in the same state" — **BUT alpha**, and the killers all hold in 2026: **"background processes launched by a call to `Sandbox.exec` will not be properly restored,"** "open TCP connections will be closed," "restored from Snapshots cannot run with GPUs," **"snapshotting a sandbox will currently cause it to terminate,"** snapshots **"expire 7 days after creation"** and **"cannot currently be extended."** API: `_experimental_snapshot()` / `Sandbox._experimental_from_snapshot()`.
- The "exec background procs not restored" clause is **fatal for a desktop**, because OpenGeni's whole display stack (Xvfb→x11vnc→websockify) is launched **via `session.exec`** (see `ensureDisplayStack` → `opengeni-desktop-up`). A Modal memory-snapshot resume would bring back a box whose X stack processes are *not* reliably restored — exactly the thing hibernation is supposed to preserve.

### Blaxel — **prior analysis UNDERSTATED it** (correction)
[docs.blaxel.ai/Sandboxes/Overview](https://docs.blaxel.ai/Sandboxes/Overview), [blaxel.ai/sandbox](https://blaxel.ai/sandbox):
- Standby **automatically** on ~15s idle, snapshots **"the complete file system in memory, preserving both files and running processes,"** memory state maintained, **resume <25ms**, **indefinite standby**. Only external network connections drop.
- This is a *stronger* hibernation primitive than the prior write-up credited ("less desktop-specific evidence" is true, but "process+memory standby" is first-class and automatic). **The OpenGeni descriptor `blaxel.supportsSuspendResume:false` is now stale** (see Part 1).
- Still **no desktop/VNC evidence** in Blaxel docs — desktop would be a BYO-image concern.

### Daytona — **doc semantics genuinely inconsistent** (prior analysis correct to flag)
[daytona.io/docs/en/sandboxes](https://www.daytona.io/docs/en/sandboxes/), [snapshots](https://www.daytona.io/docs/en/snapshots/):
- One doc surface: **`stop` clears memory** ("their memory state is cleared," filesystem-only), and **`includeMemory` snapshots are described as Windows-only**.
- Another surface: **`stop()` *is* pause** and "preserves both filesystem and memory state… retains its state in memory but does not consume CPU cycles," with `force=true` being the ungraceful no-memory teardown.
- **Auto-stop fires even with running processes** ("triggers even if there are internal processes running"), default 15 min, **`0` disables it**. Background scripts do NOT reset the idle timer.
- Native **VNC Access (Human Tools) + Computer Use (Agent Tools)**.
- Net: Daytona *can* hibernate a Linux desktop, but the memory-preservation contract is **not reliably documented for Linux**, and **auto-stop can fire under a live desktop** — both real risks the prior analysis named correctly.

### Vercel / Runloop / Cloudflare — filesystem-only (prior analysis correct)
No process/RAM resume primitive. Vercel Sandbox is FS-checkpoint + 5h hard lifetime; the descriptor already encodes `vercel.snapshot.kind:"tar-only"`, `hardLifetimeMs: 5h`.

**Bottom line on the ranking:** E2B remains #1 for *desktop* hibernation (process+memory pause/resume **and** a first-party noVNC desktop). The correct #2 is now **Blaxel** for the *hibernation primitive* (automatic process+memory standby, <25ms, indefinite) — held back only by lack of a desktop image, not by the suspend/resume mechanism. Daytona is #3 (desktop yes, memory-on-Linux unreliable + auto-stop hazard). Modal is a distant also-ran for *desktop* hibernation specifically, despite being OpenGeni's current default desktop backend.

## Part 1 — Does OpenGeni already model suspend/resume + snapshot, and how does Modal use it now?

**Yes, the capability descriptors already model it richly — but the current Modal integration deliberately does NOT use memory hibernation. It uses filesystem-persistence + cold-recreate.**

### The descriptor fields exist
`packages/contracts/src/index.ts:68-79` defines exactly the fields the goal asks about:
```
lifetime: { hardLifetimeMs?, requiresSnapshotRollover, hasIdleKiller,
            supportsSuspendResume, resumeIsLockFree, idleKillDisableHint? }
snapshot: { kind: "native-fs"|"native-dir"|"native-snapshot-id"|"tar-only"|"none", hasTarFallback }
```
Per-backend values (`index.ts:101-403`):
- **modal** (`:114-121`): `supportsSuspendResume:true`, `resumeIsLockFree:true`, `requiresSnapshotRollover:true`, `hardLifetimeMs:24h`, `hasIdleKiller:true`, `snapshot.kind:"native-fs"`.
- **e2b** (`:190-196`): `supportsSuspendResume:true`, `resumeIsLockFree:false`, `requiresSnapshotRollover:false`, `snapshot.kind:"native-snapshot-id"`, `hasIdleKiller:true`.
- **blaxel** (`:215-221`): `supportsSuspendResume:false` ← **stale per Part 0**, `snapshot.kind:"tar-only"`.
- **daytona** (`:140-146`): `supportsSuspendResume:true`, `resumeIsLockFree:false`, `snapshot.kind:"native-snapshot-id"`, `hasIdleKiller:true`.

### How "resume" is wired today
`supportsSuspendResume`/`resumeIsLockFree` drive the **warm-reattach-by-id** branch in `establishSandboxSessionFromEnvelope` (`packages/runtime/src/sandbox/index.ts:730-757`): if the envelope carries a resumable instance id, it deserializes and calls `client.resume(state)` (Modal `fromId` — no lock, R4-safe). On a provider **NotFound only**, it cold-restores via `create()` (`:745-754`). `resumeIsLockFree` is the doctrine that Modal's `fromId` reattach needs no spawn lock; a non-lock-free provider (e2b/daytona `reconnect`) still resumes by id but the system leans harder on the lease CAS.

### But the current persistence model is filesystem-only, NOT memory hibernation
This is the central finding. The reaper (`apps/worker/src/activities/sandbox-lease.ts:431-612`, `terminateProviderBox`) on drain does:
1. resume the box by id,
2. **`session.persistWorkspace()`** — snapshot **only `/workspace`** (a Modal FS-snapshot-ref or a tar archive) (`:519-537`),
3. fold the archive onto the lease under the epoch fence (`:551-583`),
4. **`client.delete()` — terminate the box** (`:588-603`).

On the next acquire, `establishSandboxSessionFromEnvelope`'s `coldRestore` (`index.ts:675-708`) **`create()`s a fresh box** and replays the archive via `hydrateWorkspace`. So between idle periods OpenGeni **kills the machine and recreates it from a `/workspace` tarball**. Everything the hibernation goal wants preserved — the X/Xvfb/x11vnc/websockify processes, the browser process tree and its tabs/JS heap, terminals, dev servers listening on ports, in-memory state — **is destroyed and rebuilt cold.** The X stack is re-launched fresh by `ensureDisplayStack` (`sandbox-resume.ts:359`), the dev servers are gone unless something in `/workspace` restarts them.

`requiresSnapshotRollover:true` on Modal refers to the **24h hard-lifetime rollover** (re-key under a new epoch before the ceiling), and `effectiveModalIdleTimeoutSeconds` (`providers/modal.ts:37`) pins Modal's idle timeout to the hard lifetime so **OpenGeni's reaper — not Modal's idle-reap — governs teardown** and gets a chance to snapshot `/workspace` first. This is a deliberate "we control the teardown, we capture the FS" design — it just isn't *memory* hibernation.

**So:** the descriptors model suspend/resume, and resume-by-id is real, but the drain/re-warm cycle is FS-persistence + cold-recreate. Memory/process hibernation is **modeled in the type system but not exercised by any backend today.**

## Part 2 — What an E2B-pause/resume desktop-hibernation backend looks like

E2B already exists as a backend (`providers/e2b.ts`, descriptor `index.ts:178-202`), but it is wired into the **same FS-persistence + terminate-on-drain reaper as Modal**. A true hibernation backend is less "an 11th enum member" and more **"a second lifecycle mode the reaper/resume honor for an existing backend"**, gated by the descriptor. Concretely:

### Descriptor fields (the contract changes)
For an `e2b` (or a new `e2b-desktop`) row tuned for hibernation:
- `supportsSuspendResume: true` — already set.
- **`resumeIsLockFree: false`** — E2B `connect()` needs the lease CAS to fence double-resume (no `fromId`-style lock-free reattach guarantee); keep as-is.
- **`snapshot.kind`** — the current `"native-snapshot-id"` is the **wrong abstraction for hibernation**. The existing four kinds all describe a *filesystem* artifact replayed onto a *fresh* box. Hibernation needs a new kind, e.g. **`"native-suspend"`** (or a `lifetime.suspendMode: "memory" | "filesystem"` discriminator), meaning: *resume is a process-state restore of the SAME logical box, not a create()+hydrate of a new one.* This is the single most important new field — it tells the reaper "**pause, don't terminate**" and tells the resume path "**connect/resume restores processes; do NOT cold-recreate or re-run `ensureDisplayStack` from scratch.**"
- **`hasIdleKiller: true` + `idleKillDisableHint`** — E2B's idle action is configurable: the config already exposes `e2bTimeoutAction: ["pause","kill"]` (`config:369`). For hibernation set **`timeoutAction:"pause"`** so the provider's own idle-killer *pauses-not-destroys* — this is the provider-native equivalent of OpenGeni's drain, and it must be reconciled with the reaper (Part 3).
- **`requiresSnapshotRollover`** — set **`true`** for E2B-hibernation because of the **30-day paused-snapshot TTL** (Part 0). A box paused longer than 30 days is GC'd by E2B; OpenGeni must roll it over (re-pause to refresh, or cold-restore from the last `/workspace` archive) before the cliff — directly analogous to the Modal 24h rollover the system already handles. The current descriptor has `requiresSnapshotRollover:false` for e2b, which is **wrong once you rely on pause-snapshots for state** — this should flip.

### Provider seam (`ProviderRegistration`, `providers/types.ts:23-38`)
The seam is already correct; the work is in the **runtime verbs the seam's client exposes**, which the leaf calls structurally:

1. **`create`** — unchanged: `E2BSandboxClient` with `exposedPorts:[6080,7681]` merged at `index.ts:264-289` (E2B is `preview-url` exposure, not on-demand, so it pre-declares the desktop+ttyd ports). Boot the `@e2b/desktop` image so noVNC/websockify exist.

2. **`resume`** — today maps to E2B `connect()` and resumes a paused sandbox **with its process tree intact**. The branch at `index.ts:730` already calls `client.resume(state)`; for hibernation this is the *whole game* — `connect()` to a paused box returns it running, X server and browser tabs alive. The **key change is upstream of resume**: the reaper must have *paused* rather than *terminated* it (Part 3).

3. **`serializeSessionState` / `deserializeSessionState`** (`index.ts:316-319`, `:484-509`) — for hibernation these carry the **paused-sandbox id** (E2B's `sandboxId` survives pause), not a `/workspace` archive. The envelope's `providerState.sandboxId` (the `hasResumableInstance` gate at `index.ts:720-727`) is what makes the warm-reattach branch fire. Hibernation **leans entirely on this id-based resume** and should NOT populate `sessionState.workspaceArchive` (the FS-replay field read by `readWorkspaceArchiveFromEnvelopeSessionState`, `index.ts:393`) — the memory snapshot *is* the persistence; a parallel FS tarball is redundant and would trigger the cold-recreate path.

4. **`resolveExposedPort`** (`stream-port.ts:42-44`, called at `:154`) — E2B exposes ports as **preview URLs** (`portExposure.kind:"preview-url"`). On resume, the preview host **may be the same or may rotate**; `resolveExposedPort(6080)` re-resolves it either way and `buildStreamUrl` reassembles the WS URL. This already works structurally — the hibernation backend needs no new port logic, only to ensure `resolveExposedPort` returns the *post-resume* preview URL (E2B's `getHost(6080)`).

5. **`persistWorkspace` — should be a no-op or absent** for a hibernation backend. The reaper at `sandbox-lease.ts:519-521` calls `session.persistWorkspace()`; if the hibernation client doesn't implement it, the no-archive path (`:571-583`) runs a CAS-check and proceeds. But for hibernation the reaper shouldn't reach `client.delete()` at all (Part 3) — `persistWorkspace` being absent is correct because **the pause itself is the snapshot.**

### The one structural addition
A `suspend(state)` verb on the client (E2B `pause()`), distinct from `delete(state)` (E2B `kill()`). Today the reaper only knows `delete`. Hibernation needs the reaper to call **`suspend` when `snapshot.kind==="native-suspend"`** and `delete` otherwise. This is the cleanest seam: one new optional client method + one descriptor-driven branch in `terminateProviderBox`.

## Part 3 — The hard part for a desktop: noVNC reconnect + lease/idle-killer interplay

### (a) Does the existing display-stack / stream-port handle a paused+resumed box with a new tunnel? **Partially — the reconnect plumbing exists, but it assumes a *fresh* X stack, not a *resumed* one.**

What already works in OpenGeni's favor:
- **Tunnel re-resolution is per-warm, not cached forever.** The spawner records `data_plane_url` on the lease (`sandbox-resume.ts:419-441`); on a rollover the **epoch advances and the slow path re-records a fresh `dataPlaneUrl`** (`viewer.ts:433-458`). `mintDesktopStream` fast-paths a cached URL only *within an epoch* and re-resolves on a new epoch. So **a resumed box that comes back on a new preview URL is handled** — provided the resume advances the lease epoch.
- **Viewer reconnect is already event-driven.** `mintDesktopStream` emits a **`stream.url.rotated` Channel-A event** on an epoch advance (`viewer.ts:345-354`) so other connected noVNC viewers reconnect to the new URL. The `StreamTokenPayload.leaseEpoch` fence (`contracts:681-695`) kills stale tokens pointing at the dead tunnel. **This is exactly the "external clients reconnect — that is fine" behavior the goal accepts.** The browser's noVNC client just re-dials the new WS URL; the RFB session re-handshakes against the (resumed) x11vnc.
- The display stack is launched over `exec` and is **flock-idempotent + re-establishes after a rollover** (`display-stack.ts`, `sandbox-resume.ts:254-263`).

What does **NOT** work for true hibernation, and is the core tension:
- The whole codebase is written around **`ensureDisplayStack` re-launching a FRESH Xvfb/x11vnc/websockify on resume** (`sandbox-resume.ts:13-15`: "re-establishes after a snapshot rollover… after a snapshot rollover dropped the X stack"). The fast lock-free pre-check (`display-stack.ts:128-136`) `nc -z`'s ports 6080/5900 and **short-circuits if x11vnc is already up** — so on a *genuine* hibernation resume (E2B/Blaxel, where x11vnc survives) it correctly no-ops and the **live framebuffer with the browser's open tabs is preserved.** Good. But the surrounding doctrine *expects* the stack to be gone and *expects* to rebuild it. For a hibernation backend the invariant must flip to: **"on resume the X stack and browser tree are ALREADY up; ensureDisplayStack must be a pure verification no-op, and re-launching it would be a regression (it would kill the very tabs we hibernated)."** The pre-check makes this safe *today*, but it is incidental, not designed — a hibernation backend should assert it.
- **The framebuffer continuity caveat (E2B-specific):** after `auto_paused` resume, the X/websockify *ports* are listening but processes aren't enumerable (Part 0). The `nc -z` pre-check keys on **ports**, so it correctly detects "up" — this is actually the *right* signal and dodges the `commands.list` bug. Worth pinning as the canonical readiness probe for hibernation backends.

### (b) Lease/idle-killer interplay — **the central architectural conflict, and it's a real one**

OpenGeni's reaper (`sandbox-lease.ts`) is built on **"drain → terminate → cold-recreate."** A provider that **pauses-not-destroys** on idle inverts a load-bearing assumption in three places:

1. **`isProviderSandboxNotFoundError` (`index.ts:561-605`) — a paused box is NOT NotFound.** The reaper's resume-by-id (`sandbox-lease.ts:500`) and the turn's `establishSandboxSessionFromEnvelope` (`index.ts:739`) treat **only NotFound** as license to cold-restore. A **paused** E2B/Daytona box that returns a "paused"/"not running" status on a naive resume could be mis-classified. The `goneMarkers` list (`index.ts:585-598`) includes `"no longer running"` — **a paused box that reports "not running" would be falsely cold-restored (create a rival, throwing away the hibernated state).** A hibernation backend MUST make `connect()`/resume **transparently un-pause** a paused box (E2B `connect()` does resume a paused sandbox), so the reaper never sees a paused-but-gone state. If any provider surfaces "paused" as an error, the discriminator needs a **`paused` ≠ `gone`** arm. This is a concrete bug-risk the current matcher would hit.

2. **Provider idle-kill vs OpenGeni drain — who pauses?** Two valid designs:
   - **OpenGeni-driven:** keep `e2bTimeoutAction` pinned away from the cliff (as Modal's idle timeout is pinned to hard-lifetime, `providers/modal.ts:37`) and let **the reaper pause** at refcount 0 — i.e. `terminateProviderBox` calls **`suspend()` instead of `delete()`** when `snapshot.kind==="native-suspend"`. This keeps OpenGeni's clean drain-grace + epoch-fence + warm-second metering (`sandbox-lease.ts:155-209`) and just swaps the terminal action. **Recommended** — it reuses the entire reaper CAS machinery.
   - **Provider-driven:** set `timeoutAction:"pause"` and let E2B auto-pause on its own idle. Simpler, but **OpenGeni's lease would think the box is `warm` while the provider has paused it** — the lease state machine and the provider state would diverge, and warm-second metering would over-bill a paused box. This needs the reaper to reconcile lease-liveness against provider-status, which it does not do today. **Not recommended without reconciliation.**

3. **`draining→cold` semantics change.** Today `confirmDrainCold` (`sandbox-lease.ts:387`) means "box terminated, lease cold, next acquire cold-recreates." For hibernation, the equivalent end-state is **`draining→suspended`**: the box is *paused, not gone*, and the lease's `resumeState.providerState.sandboxId` still points at a **resumable** paused box. The next acquire should hit the **warm-reattach branch** (`index.ts:730`, `connect()`), NOT `coldRestore`. This argues for a **new lease liveness `suspended`** (or treating `cold`-with-a-resumable-id as "suspended") so `establishSandboxSessionFromEnvelope` resumes-by-id rather than create()+hydrate. The `hasResumableInstance` gate (`index.ts:720-727`) is *already* the right discriminator — a suspended lease keeps the `sandboxId` in the envelope, so the existing code would *try* resume first. The gap is the reaper **deleting** the box and the cold-restore path **dropping** the id; fix both and the resume path mostly already does the right thing.

4. **`requiresSnapshotRollover` + 30-day cliff as a real idle-killer.** E2B's 30-day paused-TTL and Modal's 7-day memory-snapshot expiry are themselves a form of idle-killer the lease must respect. A box "hibernated" for >30 days is gone; the reaper or a rollover sweep must **re-pause (refresh the snapshot) before the TTL**, or fall back to the `/workspace` FS archive for a true cold start. The descriptor's `requiresSnapshotRollover` + `hardLifetimeMs` are the right place to encode this; the **rollover machinery the system already runs for Modal's 24h ceiling is the template.**

### Summary judgment
- **Reconnect (a): ~80% there.** Epoch-fence + `stream.url.rotated` + per-epoch `resolveExposedPort` re-resolution + the `nc -z` port pre-check already give a clean noVNC reconnect to a resumed box on a possibly-new tunnel. The one design flip required: **for hibernation backends, `ensureDisplayStack` must be a verify-only no-op (the X stack and tabs survive the pause); never re-launch it.**
- **Lease/idle interplay (b): the real work.** The reaper is hardcoded to **terminate**; hibernation needs a **`suspend` action + a `suspended` end-state + a `paused≠gone` discriminator + a snapshot-TTL rollover**. None of this is exotic — it reuses the existing drain-grace, epoch-fence, CAS, and rollover machinery — but it is the load-bearing change. The cleanest shape is **descriptor-gated** (`snapshot.kind:"native-suspend"` / a `suspendMode` field) so a single branch in `terminateProviderBox` (`sandbox-lease.ts:588`) chooses `suspend()` over `delete()`, and the `coldRestore` fallback (`index.ts:675`) is reached only on a genuine NotFound or a TTL-expired snapshot.

### Files that change, concretely
- `packages/contracts/src/index.ts:76-79` — add `snapshot.kind:"native-suspend"` (or `lifetime.suspendMode`); flip `e2b.requiresSnapshotRollover→true`, set `e2b.snapshot.kind` accordingly; **correct `blaxel.supportsSuspendResume→true`** (Part 0).
- `packages/runtime/src/sandbox/index.ts:561-605` — `paused≠gone` arm in `isProviderSandboxNotFoundError`; `index.ts:675-764` — gate `coldRestore` so a suspended-resumable envelope resumes-by-id and only falls to create() on true NotFound / TTL-expiry.
- `apps/worker/src/activities/sandbox-lease.ts:431-612` — descriptor-driven `suspend()` vs `delete()`; skip `persistWorkspace` when the pause is the snapshot; new `draining→suspended` (or cold-with-resumable-id) commit.
- `packages/runtime/src/sandbox/display-stack.ts` / `apps/worker/src/sandbox-resume.ts:359` — assert verify-only on hibernation resume; never re-launch the X stack over a surviving one.
- `packages/config/src/index.ts:369` — default `e2bTimeoutAction:"pause"` for the hibernation profile (with the OpenGeni-driven-pause caveat from (b)(2)).

The architecture is well-positioned: the descriptor already has the vocabulary, resume-by-id already exists, and the noVNC reconnect/epoch-fence is already event-driven. The gap between today's **FS-persistence + cold-recreate** and true **desktop hibernation** is a focused, descriptor-gated swap of the reaper's terminal action from *terminate* to *suspend*, plus making the resume/discriminator paths treat *paused* as *resumable* rather than *gone*.

## Per-provider structured findings

### E2B

**preservationClass:** process-ram-snapshot

**desktopSupport:** First-party, mature. E2B ships a dedicated Desktop Sandbox SDK (e2b-dev/desktop, packages e2b-desktop (Python) and @e2b/desktop (TS); latest @e2b/desktop@2.3.1, 2026-06-08; desktop Python SDK v2.0.1). Stack = Ubuntu + Xvfb virtual framebuffer on display :0 (1024x768x24, started `Xvfb :0 -ac -screen 0 1024x768x24 -retro -dpi 96 -nolisten tcp -nolisten unix`) + XFCE4 (startxfce4) + x11vnc (port 5900, nopw/forever) + noVNC + websockify on port 6080. The desktop SDK exposes desktop.stream.start()/getUrl() for the noVNC web stream, plus launch(app) (e.g. google-chrome, firefox, vscode), screenshot, mouse/keyboard control (xdotool/scrot), and bash. It is the canonical computer-use desktop used by Anthropic Computer Use / OpenAI Operator integrations and in production by Manus. This is the strongest first-party desktop story of the providers surveyed.

**whatSurvivesPauseResume:** E2B's pause is a full memory snapshot, not filesystem-only. Official doc: on pause "both the sandbox's filesystem and memory state will be saved. This includes all the files in the sandbox's filesystem and all the running processes, loaded variables, data, etc." On resume (via Sandbox.connect()) "all the running processes, loaded variables, and data are restored." Per-item for a DESKTOP sandbox: (1) X/Xvfb + XFCE + x11vnc + noVNC/websockify — these are ordinary Linux processes inside the microVM, so they are captured in the RAM snapshot and come back running (no doc explicitly walks through the desktop case, so this is inference from the generic mechanism, BUT issue #1031 directly confirms a real process — a Next.js dev server — survives pause/resume and keeps holding port 3000). (2) Browser process tree + open tabs + JS heap — STRONG case for true in-memory survival: the chrome/chromium renderer processes are part of the RAM snapshot, so the SAME renderer PIDs with the SAME in-memory-only JS heap should come back (this is NOT mere Chrome session-restore; CRIU-class memory snapshot restores the actual process memory). This specific browser-tab-heap scenario is NOT individually documented by E2B, so treat as high-confidence-but-untested-by-docs. (3) terminals/shell processes, in-memory app/DB state, loaded variables — explicitly covered ("running processes, loaded variables, data"). (4) dev servers (Vite/Next) + listening ports — survive: issue #1031 shows the dev server stays alive and port 3000 stays in use across auto-pause/resume. (5) Filesystem — preserved.

**whatBreaks:** (1) STILL BETA, not GA: the API is namespaced beta_pause()/beta_create() in the current Python SDK v2.5.0, each carrying "[BETA] This feature is in beta and may change." The polished prose on docs/sandbox/persistence overstates maturity. (2) External clients drop, by design: "If you have a service (a server) running inside your sandbox and you pause the sandbox, the service won't be accessible from the outside and all the clients will be disconnected... If you resume the sandbox, the service will be accessible again but you need to connect clients again." So the noVNC WebSocket and any external DB/WS connection close and must reconnect — acceptable per the goal. (3) Process-management tracking bug (issue #1031, OPEN, label bug/envd): after auto-pause+resume the dev server is still running and still holds its port, but is INVISIBLE to sandbox.commands.list() — the process survives but the SDK can no longer manage/list it. (4) Filesystem persistence bug across MULTIPLE cycles (issue #884, closed/Orchestrator): first pause→resume preserves changes, but subsequent cycles could lose file modifications — verify on your version. (5) autoPause/connect bug (issue #875, closed): connect() hardcodes autoPause:false in the request body, overriding the creation-time autoPause and potentially killing the sandbox on reconnect. (6) Historical "pause deleted my sandbox" (issue #157, closed/duplicate) — earlier beta_pause actually deleted boxes; appears fixed but signals beta instability. (7) Continuous-runtime cap: a sandbox can run only ~1h (base) / ~24h (Pro) of continuous uptime before it must be paused/killed; the cap resets after each pause/resume cycle — so long-lived desktops REQUIRE periodic pause. (8) No E2B blog/example demonstrates a paused-and-resumed live GUI desktop end-to-end (VNC + open browser tabs) — the desktop+pause combination is undocumented as a worked example.

**apiAndLifecycle:** Create with pause support: Sandbox.beta_create(template=..., timeout=<sec>, auto_pause=True) (Python) — auto_pause makes the box auto-pause (instead of being killed) after `timeout` seconds of inactivity. Manual pause: sbx.beta_pause() (instance) or Sandbox.beta_pause(sandbox_id) (classmethod) — JS docs surface it as sandbox.pause() but the typed SDK reference is beta_pause. Resume: there is NO resume()/beta_resume(); you call Sandbox.connect(sandbox_id) (or sbx.connect()) and a paused box is auto-resumed with full context restored. List paused boxes: Sandbox.list() with a state query param. Idle-shutdown gating: driven by the timeout parameter (idle TTL in seconds) plus auto_pause=True — when the idle timeout elapses, the box auto-pauses (full-state) rather than terminating. Lifecycle primitives: timeout (idle TTL), auto_pause (pause-vs-kill on idle), kill() (explicit teardown that deletes the persisted snapshot). ### Modal

**preservationClass:** process-ram-snapshot

**desktopSupport:** First-party desktop example exists but is NOT a managed product — it is the open-source "Run Anthropic's computer use demo in a Modal Sandbox" example (https://modal.com/docs/examples/anthropic_computer_use). You bring your own desktop stack in a Docker image: Xvfb + XFCE + x11vnc + noVNC/websockify. The desktop and Streamlit UI are surfaced via Modal tunnels using encrypted_ports=[8501, 6080] on Sandbox.create. There is NO managed "desktop sandbox" SKU and NO managed computer-use/CUA API like Daytona/E2B offer — Modal positions Sandboxes as a generic code-execution primitive, so the entire desktop/VNC layer is user-assembled and user-maintained. Maturity for desktop: example-grade, not productized.

**whatSurvivesPauseResume:** Modal has NO pause/resume/standby primitive. The only memory-preserving mechanism is the alpha "Sandbox memory snapshot" (_experimental_snapshot), which captures "a Sandbox's entire state, both in memory and on the filesystem" and restores it as a NEW sandbox. Per-item for a desktop, given the docs: (1) X/Xvfb + window manager + x11vnc + noVNC IF and ONLY IF they are the sandbox ENTRYPOINT process tree (the computer-use example runs ./entrypoint.sh as the create() command) — entrypoint/main-process state is what the snapshot is built around; (2) browser process tree + open tabs + JS heap — survives only if the browser is part of the entrypoint-launched tree, NOT if it (or the desktop) was started via Sandbox.exec; (3) terminals/shell/dev-server processes started via Sandbox.exec — DO NOT survive ("any background processes launched by a call to Sandbox.exec will not be properly restored after a snapshot"); (4) listening ports / in-memory DB — in-memory state is captured, but all OPEN TCP CONNECTIONS are closed on snapshot and must be reopened on restore (listeners come back with the process, live connections do not). Net: a desktop CAN survive only if the ENTIRE VNC+browser+dev stack is baked into one entrypoint process tree and nothing is launched via exec — which is contrary to how the noVNC example and most agent desktops actually drive the box.

**whatBreaks:** Hard gotchas, all confirmed in current docs (https://modal.com/docs/guide/sandbox-snapshots): (1) ALPHA / early-preview, gated behind _experimental_* APIs. (2) Snapshotting "will currently cause it to terminate" — taking the snapshot KILLS the original sandbox ("We intend to remove this limitation soon" — still present as of SDK 1.5.1, 2026-06-23). (3) 7-day expiry, "cannot currently be extended" — useless for long-lived hibernation; for longer persistence docs steer you to filesystem-only snapshots. (4) NO GPU: "Sandboxes created with _experimental_enable_snapshot=True or restored from Snapshots cannot run with GPUs." (5) Open TCP connections closed on snapshot, must reopen on restore. (6) Cannot snapshot while a Sandbox.exec command is still running. (7) THE CRITICAL ONE FOR DESKTOPS: background procs launched via Sandbox.exec are "not properly restored" — so a VNC/browser/dev stack started by exec is lost; it MUST be the entrypoint to survive. (8) Restore only on the SAME exact instance type. (9) Idle: Sandboxes auto-terminate via timeout (the example sets timeout=60*60 → spins down after 1h) and a separate idle_timeout on create; auto-termination is a HARD kill, not a hibernate — there is no auto-snapshot-on-idle, so idle shutdown destroys all process/RAM state unless you have proactively snapshotted (and snapshotting itself terminates the box).

**apiAndLifecycle:** There is NO pause()/resume()/standby on modal.Sandbox. Lifecycle = create / terminate / poll / wait / wait_until_ready. Memory-snapshot flow (all experimental): create with sb = modal.Sandbox.create("./entrypoint.sh", app=app, image=image, encrypted_ports=[8501,6080], _experimental_enable_snapshot=True); take snapshot with snapshot = sb._experimental_snapshot() (this TERMINATES sb); retrieve later with snapshot = modal.SandboxSnapshot.from_id(snapshot_id); restore into a brand-new sandbox with sandbox = modal.Sandbox._experimental_from_snapshot(snapshot). Filesystem-only (GA, separate, no memory): Sandbox.snapshot_filesystem(timeout=55, ttl=30*24*3600) and Sandbox.snapshot_directory(...) → both return an Image (default ttl 30 days; ttl/timeout kwargs added in SDK 1.5.0, 2026-06-09). Idle gating: create(idle_timeout=...) "amount of time in seconds that a sandbox can be idle before being terminated" and a wall-clock timeout=... both TERMINATE (not snapshot) the box; you must orchestrate snapshot-before-idle yourself, and there is no idle hook that snapshots automatically. The distinct Function-level @app.function(enable_memory_snapshot=True) + @modal.enter(snap=True) mechanism applies to Functions, not Sandboxes, and is for cold-start acceleration, not user-driven hibernation.

**retentionAndCost:** Retention: memory snapshots expire 7 days after creation, non-extendable (vs filesystem snapshots default 30-day ttl, configurable). Resume latency: docs frame snapshots around "sub-second" / 3-10x faster cold starts and "fast cold starts" via the same restore infra, but give no explicit per-GiB resume figure for sandbox MEMORY snapshots specifically. Pricing for paused/snapshot state: NOT documented on these pages — Modal exposes no "paused sandbox" billing tier (because a snapshotted sandbox is terminated, you pay $0 compute for the dead box; you pay only for snapshot storage, which is not separately priced in the snapshot docs). There is effectively no "paused but warm" cost model — it is terminate + cold-restore, so the cost question is moot but so is the latency benefit of a true standby.

**fitForExactDesktopHibernation:** constrained — the primitive exists and does preserve RAM/process state, but alpha status + 7-day cap + snapshot-terminates-original + no-GPU + exec-launched-procs-not-restored make a continuously-hibernating desktop impractical without forcing the entire VNC+browser+dev stack into a single entrypoint and re-snapshotting at least weekly.
